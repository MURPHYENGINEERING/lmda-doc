Naming convention

MACRO_NAME ENUM_ITEM
StructName TypeName
PointerType pointerVariable
localVariable parameterVariable
constantVariable _staticVariable
private_function() lm_public_function()

With the special case of a function that's declared in one module (the interface module) and defined in another (the implementation module): lmda_interface_function_impl(). Here the prefix lmda indicates the broadest scope, and the suffix impl indicates the separated declaration and definition.

Application Design

At the first level, the game uses a plugin architecture. The application hosts two plugins: the game plugin simulates the game world, and the presentation plugin represents the game world to the outside world.

There are two presentation plugins: the SDL plugin implements a 2D game, and the server plugin implements a headless server with a browser-accessible dashboard. The server is headless because it still runs the game plugin, meaning it still constitutes a game client playing the game; but the only inputs and outputs are from and to the network and AI.

Project Structure

lmda/
docs/ Contents of lmda.murphysys.com
cmake/ Build utilities, e.g. detecting architecture
src/
common/ Functionality not unique to any module, e.g. logging,
networking, filesystem, etc.
input/ Platform-independent input system with interfaces for
platform-depenendent sourcing (e.g. from SDL).
net/ Client/server TCP connections
platform/ Declarations of platform-specific functionality that will
be implemented in another module.
platform/ Definitions of platform-specific functionality from the
common module. One of these is statically linked with common.
mac/ MacOS implementations
win/ Windows 10-11 implementations (and maybe earlier)
game/ Implements the game plugin, constructing and simulating the
game world (strictly non-graphically; this plugin is common
to clients and headless servers).
sdl/ Implements the presentation plugin as a 2D top-down game
using SDL2 for graphics, audio, and input.
app/ Dynamically loads the game and presentation plugins,
provides them with persistent state, and updates them in a
rate-controlled loop. Plugins can be hot-reloaded to update
code in-place without affecting state, which resides
in the application.
